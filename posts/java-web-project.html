System Architecture

MVC Pattern - Model View Controller

FishStore is built on the standard MVC model, ensuring clear separation between layers, making it easy to maintain and expand in the future. This architecture helps the team work in parallel without conflicts and makes the system highly scalable.

View Layer (Presentation)

Technologies used:

  • JSP + JSTL: Dynamic content rendering without Java code in HTML
  • HTML5/CSS3: Modern, semantic interface
  • JavaScript: Client-side interaction, form validation
  • Responsive Design: Works on all devices
  • Font Awesome: Professional icon system

Responsibilities:

  • Display product lists, details
  • Render shopping cart, order forms
  • Display user messages and notifications
  • Handle UI interactions

Controller Layer (Business Logic)

Main Servlets:

  • HomePageServlet (/home): Display product list, search, filter by category/price
  • AdminServlet (/admin): Complete CRUD for products, manage categories, view statistics
  • AuthenServlet (/authen): Login, logout, register, session management
  • CartServlet (/cart): Add/remove/update quantity in cart
  • AIChatServlet (/ai-chat): Process AI requests, integrate Gemini API

Filters:

  • AuthFilter: Check login status
  • AdminFilter: Verify admin permissions

Model Layer (Data Access)

DAO Pattern implementation:

  • KhachHangDAO: User CRUD, authentication
  • LoaiCaDAO: Product management (fish types)
  • DanhMucDAO: Category management
  • GioHangDAO: Shopping cart operations
  • DonHangDAO: Order & order details management

Supporting utilities:

  • DBContext: Connection pooling, database connection
  • PasswordUtil: BCrypt password encryption
  • ValidationUtil: Input validation
  • GeminiService: Connect to Google Gemini AI

Request Flow in System

Here's a detailed example of how the system processes a product purchase request:

  1. User clicks "Add to Cart" on home.jsp page
  2. Request sent to CartServlet via POST method with product_id and quantity
  3. CartServlet validates: Check login session, check product_id validity
  4. CartServlet calls GioHangDAO: Add product to cart or update quantity if already exists
  5. GioHangDAO executes SQL: INSERT or UPDATE in GIO_HANG table
  6. CartServlet retrieves updated cart: Get all items in current user's cart
  7. Forward to cart.jsp: Display cart with all items
  8. cart.jsp renders: Show list using JSTL forEach, calculate total

Security Architecture

Authentication & Authorization

  • Password encryption: Using BCrypt with salt, no plain text storage
  • Session management: HttpSession for user state
  • Role-based access: Customer, Staff, Admin with different permissions
  • Filter protection: AuthFilter & AdminFilter check every request

Attack Prevention

  • SQL Injection: Using PreparedStatement throughout
  • XSS Protection: Input sanitization, output encoding in JSP
  • CSRF Prevention: Token validation for important forms
  • Input Validation: Validate all user input before processing

Data Security

  • Connection pooling: Prevent connection leak
  • Transaction management: Ensure data consistency
  • Error handling: Don't expose sensitive information
  • Logging: Record all important operations

Database Design

Relational Database Model

The system uses SQL Server with standardized relational tables, ensuring data integrity and optimal performance. Database design follows 3NF principles to minimize redundancy and ensure consistency.

KHACH_HANG (Customer)

Key fields:

  • ma_khach_hang (PK, IDENTITY)
  • ten_dang_nhap (UNIQUE, NOT NULL)
  • mat_khau (BCrypt hashed)
  • ho_ten, email, so_dien_thoai
  • vai_tro (Admin/Staff/Customer)
  • trang_thai (Active/Inactive)

Purpose: Store all users including customers, staff, and admin in one table with role differentiation

DANH_MUC (Category)

Key fields:

  • ma_danh_muc (PK, IDENTITY)
  • ten_danh_muc (NOT NULL)
  • mo_ta (Description)
  • trang_thai (Active/Inactive)

Purpose: Classify products (Freshwater fish, Sea fish, etc.)

LOAI_CA (Fish Type - Product)

Key fields:

  • ma_loai_ca (PK, IDENTITY)
  • ma_danh_muc (FK to DANH_MUC)
  • ten_loai_ca, mo_ta
  • don_vi_tinh (unit: Kg)
  • gia_ban (DECIMAL 18,2)
  • hinh_anh (image filename)
  • ngay_tao, trang_thai

Purpose: Store detailed product information

GIO_HANG (Shopping Cart)

Key fields:

  • ma_gio_hang (PK, IDENTITY)
  • ma_khach_hang (FK)
  • ma_loai_ca (FK)
  • so_luong (DECIMAL 10,2)
  • gia_tam_tinh (temporary price)
  • ngay_tao

Purpose: Store items in user's cart before checkout

DON_HANG (Order)

Key fields:

  • ma_don_hang (PK, IDENTITY)
  • ma_khach_hang (FK)
  • ma_don_hang_hien_thi (display ID)
  • ten_nguoi_nhan, so_dien_thoai_nhan
  • dia_chi_giao_hang
  • tong_tien_hang, phi_van_chuyen
  • tong_thanh_toan
  • phuong_thuc_thanh_toan (COD/Transfer)
  • trang_thai (Pending/Confirmed/Shipping/Delivered)

Purpose: Store order header information

CHI_TIET_DON_HANG (Order Detail)

Key fields:

  • ma_chi_tiet_dh (PK, IDENTITY)
  • ma_don_hang (FK)
  • ma_loai_ca (FK)
  • so_luong, don_gia, thanh_tien

Purpose: Store line items of each order (which products, quantity, price)

Relationships Between Tables

  • KHACH_HANG ‚Üí GIO_HANG: One-to-Many (One customer has many cart items)
  • KHACH_HANG ‚Üí DON_HANG: One-to-Many (One customer has many orders)
  • DANH_MUC ‚Üí LOAI_CA: One-to-Many (One category has many fish types)
  • LOAI_CA ‚Üí GIO_HANG: One-to-Many (One fish type can be in many carts)
  • LOAI_CA ‚Üí CHI_TIET_DON_HANG: One-to-Many (One fish type can appear in many order details)
  • DON_HANG ‚Üí CHI_TIET_DON_HANG: One-to-Many (One order has many line items)

Performance Optimization

Indexing Strategy

  • Primary Keys: Clustered index on all ID columns
  • Foreign Keys: Non-clustered index for JOIN performance
  • Search columns: Index on ten_loai_ca, ten_dang_nhap
  • Status columns: Index on trang_thai for filtering

Connection Pooling

  • HikariCP implementation: Fast, lightweight connection pool
  • Max pool size: 20 connections
  • Connection timeout: 30 seconds
  • Idle timeout: 600 seconds (10 minutes)

Data Integrity

  • Foreign Key constraints: Prevent orphaned records
  • NOT NULL constraints: Ensure required fields
  • UNIQUE constraints: Prevent duplicates
  • CHECK constraints: Validate data ranges
  • Transaction management: ACID compliance

Challenges & Solutions

During 3 days of intensive development, the team encountered many technical and practical challenges. Here's how we overcame them:

Challenge 1: Integrating AI with Real Data

Problem:

  • How can AI consult based on real-time inventory instead of generic information?
  • AI doesn't have direct database access
  • Need to provide product context without exposing sensitive data

Solution:

  • Built GeminiService that queries database before calling API
  • Format product data into structured prompt for AI
  • Cache frequently accessed data to reduce database load
  • Implement rate limiting to prevent API quota exhaustion

Result: AI can now provide accurate consulting based on actual inventory with 95% accuracy

Challenge 2: Session & Authentication Management

Problem:

  • How to secure user information and manage sessions effectively?
  • Prevent unauthorized access to admin pages
  • Handle concurrent logins from same user

Solution:

  • Implemented AuthFilter to check authentication on every request
  • Created AdminFilter for role-based authorization
  • Used HttpSession with secure flags (httpOnly, secure)
  • Implemented session timeout after 30 minutes of inactivity
  • BCrypt password hashing with salt for security

Result: Zero security breaches, smooth authentication flow

Challenge 3: Shopping Cart State Management

Problem:

  • Should cart data be stored in session or database?
  • What if user closes browser before checkout?
  • How to handle cart for non-logged-in users?

Solution:

  • Hybrid approach: Session for guest users, database for logged-in users
  • Persist cart to GIO_HANG table for registered customers
  • Merge session cart with database cart on login
  • Auto-clear abandoned carts after 7 days

Result: Users don't lose cart items, improved conversion rate

Challenge 4: Code Organization & Maintainability

Problem:

  • 3 team members working simultaneously can cause conflicts
  • Need clear code structure for easy maintenance
  • Avoid code duplication across servlets

Solution:

  • Strictly follow MVC pattern with clear separation
  • Implement DAO pattern for all database operations
  • Create utility classes for common functions
  • Use Git with feature branches for version control
  • Code review before merging to main branch

Result: Clean, maintainable codebase with minimal merge conflicts

Challenge 5: Responsive Design

Problem:

  • How to make interface work perfectly on all devices?
  • Product images have different sizes/ratios
  • Cart table too wide on mobile screens

Solution:

  • Mobile-first CSS approach with media queries
  • Flexbox and CSS Grid for flexible layouts
  • Image standardization (aspect ratio 1:1, max 800x800px)
  • Convert table to card layout on small screens
  • Touch-friendly buttons (min 44x44px touch targets)

Result: 100% responsive, tested on iOS, Android, desktop

Challenge 6: Performance Optimization

Problem:

  • Product list page loads slowly with many images
  • Database queries becoming slow with test data
  • AI responses sometimes take too long

Solution:

  • Pagination: Display 12 products per page instead of all
  • Image optimization: Compress images, lazy loading
  • Database indexing: Add indexes on frequently queried columns
  • Query optimization: Use JOIN instead of multiple queries
  • Connection pooling: Reuse database connections
  • AI timeout: Set 10-second timeout for AI responses

Result: Page load time reduced from 5s to under 2s

Key Learnings from Challenges

  • Planning is crucial: Spending time on architecture design saves debugging time later
  • Security first: Think about security from the beginning, not as an afterthought
  • Test frequently: Test on different browsers and devices throughout development
  • Communication: Daily standups and clear task division prevent conflicts
  • Documentation: Document complex logic helps team members understand code
